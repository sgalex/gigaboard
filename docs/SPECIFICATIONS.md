# Требования и спецификации

## Функциональные требования (MVP)

### FR-1: Бесконечное полотно с узлами (SourceNode/ContentNode/WidgetNode/CommentNode)
**Описание**: Пользователь создаёт борд с бесконечным канвасом, размещает и перемещает узлы четырёх типов:
- **SourceNode** 🆕: источники данных (файлы, СУБД, API, промпты, streaming)
- **ContentNode** 🆕: результаты обработки (текст + N таблиц)
- **WidgetNode**: визуализации ContentNode (графики, таблицы, метрики)
- **CommentNode**: комментарии к любым узлам
**Предусловия**: Авторизованный пользователь, созданный борд.
**Шаги**:
1. Создать борд
2. Добавить SourceNode (загрузить файл, настроить БД/API, ввести промпт)
3. Извлечь данные (extract) → создаётся ContentNode
4. Создать визуализацию → создаётся WidgetNode
5. Переместить/изменить размер узлов
**Ожидаемый результат**: Узлы сохраняются и отображаются на канвасе, их позиции синхронизируются между клиентами. Чёткий data lineage от источника к визуализации.

### FR-2: CRUD бордов, узлов и трансформаций
**Описание**: Управление бордами, узлами (SourceNode/ContentNode/WidgetNode/CommentNode), связями и трансформациями через REST API.
**Предусловия**: Авторизация по JWT.
**Шаги**:
1. Создать/обновить/удалить борд
2. Создать/обновить/удалить узлы:
   - SourceNode: настроить источник данных
   - ContentNode: через extract или transform
   - WidgetNode: визуализация ContentNode
   - CommentNode: аннотации
3. Создать/удалить связи между узлами (TRANSFORMATION, VISUALIZATION, COMMENT, REFERENCE, DRILL_DOWN)
4. Создать/выполнить трансформацию (ContentNode(s) → ContentNode)
5. Извлечь/обновить данные из SourceNode (extract/refresh)
**Ожидаемый результат**: Данные сохраняются в БД; REST возвращает актуальное состояние борда/узлов/трансформаций с полным data lineage.

### FR-3: Real-time совместное редактирование
**Описание**: Клиенты получают события о добавлении, перемещении и удалении узлов (SourceNode/ContentNode/WidgetNode/CommentNode), а также о создании/удалении связей, выполнении трансформаций и обновлении streaming источников в режиме реального времени.
**Предусловия**: Пользователи подключены к комнате boardId через Socket.IO.
**Шаги**:
1. Клиент отправляет событие изменения (узел/связь/трансформация/extract/refresh)
2. Сервер транслирует событие в комнату boardId
3. Остальные клиенты применяют обновление
4. Для streaming источников: автоматические обновления ContentNode транслируются всем клиентам
**Ожидаемый результат**: Все клиенты видят синхронное состояние канваса без перезагрузки, включая real-time обновления от streaming источников.

### FR-4: AI-помощник (GigaChat)
**Описание**: Пользователь задаёт вопрос на естественном языке, система создаёт:
- **SourceNode** с конфигурацией источника (если нужен новый источник данных)
- **ContentNode** с данными (результат извлечения/анализа/трансформации)
- **WidgetNode** с визуализацией (HTML/CSS/JS код, сгенерированный WidgetCodexAgent)
- **CommentNode** с аналитическими инсайтами
**Предусловия**: Доступ к GigaChat через langchain-gigachat.
**Шаги**:
1. Отправить вопрос /ai/query
2. Planner анализирует запрос, определяет нужны ли новые источники данных
3. Researcher создаёт SourceNode (если нужно) и извлекает данные → ContentNode
4. Transformation Agent выполняет трансформации → новые ContentNode
5. WidgetCodexAgent анализирует ContentNode, генерирует визуализацию → WidgetNode
6. Узлы размещаются на борде со связями (TRANSFORMATION, VISUALIZATION)
**Ожидаемый результат**: Пользователь видит полный pipeline от источника до визуализации на доске с явным data lineage; события приходят всем участникам борда.

### FR-5: История и версии доски (минимум undo/redo локально)
**Описание**: Локальная история действий пользователя с возможностью отката последних операций на клиенте; сервер хранит последние сохранённые состояния борда.
**Предусловия**: Активная сессия на борде.
**Шаги**:
1. Совершить действие (создать/переместить/удалить)
2. Выполнить undo/redo на клиенте
3. Отправить сохранение борда на сервер
**Ожидаемый результат**: Локальный откат работает мгновенно; сохранение фиксирует итоговое состояние в БД.

### FR-6: AI Assistant Panel
**Описание**: Постоянный AI-ассистент в правом боковом панеле для диалога с пользователем. Помощник понимает контекст текущей доски (узлы, данные, связи, трансформации), предлагает рекомендации по анализу, помогает с вопросами о данных и визуализациях.
**Предусловия**: Авторизованный пользователь, открытая доска, доступ к GigaChat.
**Шаги**:
1. Открыть доску → AI Assistant Panel автоматически доступен в правой панели
2. Ввести сообщение естественном языке (вопрос, команда, запрос)
3. AI обрабатывает контекст доски (текущие узлы, данные, связи, граф трансформаций, история) и отвечает
4. Пользователь может применить предложение (создать/обновить SourceNode, ContentNode, WidgetNode или трансформацию) одной кнопкой
5. История диалога сохраняется в сессии
**Ожидаемый результат**: 
- Быстрый доступ к AI-помощи без прерывания работы
- Контекстные рекомендации на основе текущей доски и графа трансформаций
- Мгновенное создание/обновление узлов из предложений ассистента
- История сообщений доступна в текущей сессии

### FR-7: Multi-Agent System (V2)
**Описание**: За AI Assistant Panel находится мультиагентная система (Orchestrator V2), где специализированные AI агенты решают сложные задачи анализа данных через Redis Message Bus.
**Агенты** (9 core):
- Planner: Разбивает запросы на подзадачи, маршрутизирует, адаптивный replan
- Researcher: Получает данные из URL, API, веб-страниц
- Analyst: Анализирует структурированные данные → insights
- Reporter: Генерирует WidgetNode визуализации (HTML/CSS/JS)
- Validator: Dry-run валидация сгенерированного кода
- Quality Gate: Валидация результатов на соответствие требованиям
- Discovery: Поиск и обнаружение данных
- Resolver: Batch AI resolution (пол по имени, sentiment, категоризация)
- Structurizer: Извлечение структуры из текста → таблицы, сущности
**Контроллеры** (5 satellite):
- AI Assistant: управление диалогом и контекстом доски
- Transformation: генерация Python/pandas кода трансформаций
- Transform Suggestions: предложения трансформаций
- Widget: генерация визуализаций
- Widget Suggestions: предложения визуализаций на основе данных
**Коммуникация**: Orchestrator V2 через Redis Message Bus (`request_response()`)
**Ожидаемый результат**:
- Сложные многошаговые аналитические задачи решаются автоматически
- Агенты показывают свой ход мышления в чате
- Пользователь видит прогресс выполнения

### FR-8: Dynamic Tool Development
**Описание**: Developer Agent может динамически писать и выполнять инструменты (tools) для решения специфических задач. Инструменты выполняются в изолированной песочнице (sandbox) с ограничениями по ресурсам и безопасности.
**Примеры инструментов**:
- Web scraping tools: сборка данных с веб-сайтов
- Database query tools: выполнение SQL запросов к различным БД
- API integration tools: работа с REST APIs
- Data transformation tools: обработка и трансформация данных
**Требования**:
- Code generation: AI пишет код на Python/SQL/JavaScript
- Security validation: проверка кода перед выполнением
- Sandbox execution: изоляция с timeout/memory limits
- Tool versioning: хранение версий инструментов
- Tool registry: переиспользование инструментов
**Ожидаемый результат**:
- Гибкое решение новых задач без кодирования вручную
- Безопасное выполнение пользовательского кода
- Переиспользование созданных инструментов

### FR-9: WidgetNode Generation (Генерация визуализаций)
**Описание**: Reporter Agent анализирует **ContentNode** и генерирует HTML/CSS/JavaScript код для создания WidgetNode. Агент может создавать 6 типов визуализаций с поддержкой адаптивного дизайна и темного режима.

**Процесс**:
1. Reporter Agent получает **ContentNode** с данными (текст + N таблиц)
2. Анализирует схему таблиц, типы полей, размер датасета
3. Определяет оптимальный тип визуализации
4. Генерирует HTML/CSS/JS код для WidgetNode
5. Создает WidgetNode на канвасе
6. Создает edge **VISUALIZATION**: ContentNode → WidgetNode

**Типы WidgetNode**:
- **Metric**: KPI с трендом (из числовых ContentNode)
- **Chart**: линейные, столбчатые, круговые диаграммы (из табличных ContentNode)
- **Table**: сортируемые таблицы (из любых табличных ContentNode)
- **Heatmap**: 2D матрицы для корреляций (из матричных ContentNode)
- **Gauge**: индикаторы прогресса (из числовых ContentNode с диапазонами)
- **Custom HTML**: произвольный код для специфических визуализаций

**Визуализация множественных таблиц**:
- Один ContentNode может содержать N таблиц
- WidgetNode может использовать одну или несколько таблиц для комплексных визуализаций
- Множественные WidgetNode могут визуализировать один ContentNode

**Требования**:
- Код генерируется на основе анализа ContentNode (схема, типы данных, размер)
- Множественные WidgetNode могут визуализировать один ContentNode
- Валидация кода (размер, безопасность, доступность)
- Адаптивный дизайн (мобильный, планшет, десктоп)
- Поддержка темного режима через CSS переменные
- Локальный рендер в чате + размещение на доске
- **Автообновление**: при изменении ContentNode, связанные WidgetNode автоматически обновляются

**Ожидаемый результат**:
- Reporter Agent создает визуализации из ContentNode
- Пользователь видит превью WidgetNode в чате перед размещением
- WidgetNode связан с родительским ContentNode через edge VISUALIZATION
- При обновлении ContentNode (например, refresh из SourceNode), все WidgetNode автоматически обновляются

### FR-10: Node Management System (Система управления узлами)
**Описание**: AI-агенты активно строят аналитические доски, создавая узлы (**SourceNode/ContentNode/WidgetNode/CommentNode**) и семантические связи между ними. Система управляет графом зависимостей и автоматизацией трансформаций.

**Возможности агентов**:

**Работа с SourceNode**:
- Создание SourceNode (POST /api/v1/boards/{boardId}/source-nodes)
- Поддержка всех типов источников: prompt, file, database, api, stream, manual
- Операции: extract, validate, refresh, archive
- Автоматическое обновление по расписанию для api/database/stream

**Работа с ContentNode**:
- Создание ContentNode (POST /api/v1/boards/{boardId}/content-nodes)
- Загрузка контента (text + N таблиц) после extraction из SourceNode
- Обновление ContentNode при изменении SourceNode (SourceNode наследует ContentNode)
- Удаление ContentNode с каскадным удалением зависимых WidgetNode

**Работа с WidgetNode**:
- Создание WidgetNode из ContentNode (POST /api/v1/boards/{boardId}/widget-nodes)
- Генерация HTML/CSS/JS кода визуализации
- Связывание с родительским ContentNode через VISUALIZATION edge
- Автообновление при изменении ContentNode

**Работа с CommentNode**:
- Создание комментариев к SourceNode/ContentNode/WidgetNode
- AI-генерированные инсайты и аннотации

**Работа с трансформациями**:
- Создание TRANSFORMATION edges: ContentNode(s) → ContentNode
- Генерация и выполнение кода трансформаций
- Автоматический replay при обновлении source ContentNode
- Управление графом зависимостей (data lineage)

**Управление связями**:
- Создание edges всех типов (TRANSFORMATION, VISUALIZATION, COMMENT, REFERENCE, DRILL_DOWN)
- Валидация связей (проверка типов, циклические зависимости)
- Визуализация графа зависимостей

**Layout Planning**:
- Автоматическая организация узлов (flow, grid, hierarchy, freeform)
- Группировка связанных узлов
- Оптимизация расположения для читаемости графа

**История и версионирование**:
- Сохранение всех изменений доски
- Версионирование кода трансформаций
- Откат к предыдущим состояниям

**Ожидаемый результат**:
- Агенты создают и управляют узлами всех типов (SourceNode, ContentNode, WidgetNode, CommentNode)
- Пользователь видит построение доски в реальном времени
- Граф зависимостей автоматически поддерживается
- Трансформации автоматически переиспользуются при обновлении данных
- Завершенная доска содержит полный граф: SourceNode → ContentNode → WidgetNode + CommentNode

### FR-11: Connection Types & Node Relationships (Типы связей между узлами)
**Описание**: Узлы на доске связаны семантическими связями (edges), которые определяют отношения между данными, визуализациями и комментариями.

**Правила создания узлов**:
- **SourceNode** создаётся самостоятельно (загрузка файла, подключение к API, SQL, промпт, стрим, ручной ввод). SourceNode **наследует ContentNode** и содержит данные напрямую (text + tables)
- **ContentNode** создаётся через TRANSFORMATION из SourceNode/ContentNode, или независимо (AI-генерация)
- **WidgetNode** является производным от ContentNode и создаётся только при наличии `parent_content_node_id`; WidgetNode не может существовать отдельно от ContentNode. Удаление родительского ContentNode приводит к каскадному удалению/деактивации связанных WidgetNode.
- **CommentNode** создаётся только с обязательным `target_node_id` и всегда привязан к существующему узлу: **SourceNode**, **ContentNode** или **WidgetNode**. При удалении родительского узла соответствующие комментарии удаляются/деактивируются каскадно.

**Основные типы связей** (5 типов, EXTRACT удалён):

**1. TRANSFORMATION** (SourceNode/ContentNode(s) → ContentNode):
- Преобразование одного или нескольких ContentNode в новый ContentNode
- Содержит произвольный Python код трансформации
- Поддерживает множественные входы (N источников → 1 результат)
- Может автоматически переиспользоваться при обновлении источников
- Метаданные: код, prompt, execution stats, версия

**3. VISUALIZATION** (ContentNode/SourceNode → WidgetNode):
- Визуализация ContentNode через WidgetNode
- Один ContentNode может иметь множественные WidgetNode
- При обновлении ContentNode, все WidgetNode автоматически обновляются
- Метаданные: тип виджета, конфигурация визуализации
- Создаётся только при наличии `parent_content_node_id`; WidgetNode не может существовать отдельно от ContentNode. Удаление ContentNode ведёт к каскадному удалению/деактивации связанных WidgetNode.

**4. COMMENT** (CommentNode → SourceNode/ContentNode/WidgetNode):
- Комментарии и аннотации пользователей или AI
- Может содержать инсайты, аномалии, рекомендации
- Метаданные: автор (user/agent), timestamp, thread
- Создаётся только с обязательным `target_node_id`; целевой узел — **SourceNode**, **ContentNode** или **WidgetNode**. При удалении родительского узла комментарии удаляются/деактивируются каскадно.

**5. REFERENCE** (любой узел → любой узел):
- Справочная информация или зависимость
- Используется для документирования связей между узлами

**6. DRILL_DOWN** (ContentNode → ContentNode или WidgetNode → WidgetNode):
- Переход от сводных данных к детальным
- Клик на элемент показывает детализацию

**Визуализация связей**:
- **TRANSFORMATION**: синяя стрелка, пунктирная линия, иконка кода
- **VISUALIZATION**: зеленая стрелка, сплошная линия, иконка графика
- **COMMENT**: желтая стрелка, точечная линия, иконка комментария
- **REFERENCE**: серая стрелка, тонкая линия
- **DRILL_DOWN**: оранжевая стрелка, двойная линия

**Граф зависимостей (Data Lineage)**:
```
[SourceNode: CSV file]────TRANSFORMATION────>[ContentNode: Cleaned]
        │                                            │
        │                                            ├──VISUALIZATION──>[WidgetNode: Chart]
        │                                            │
        └──COMMENT──>[CommentNode: "Raw data has duplicates"]
                                                     │
                                                     └──TRANSFORMATION──>[ContentNode: Aggregated]
                                                             │
                                                             └──VISUALIZATION──>[WidgetNode: Table]
```

> **Примечание**: SourceNode наследует ContentNode, поэтому EXTRACT edge не нужен — данные находятся в самом SourceNode.

**Валидация связей**:
- Проверка типов узлов (например, TRANSFORMATION только между ContentNode)
- Обнаружение циклических зависимостей
- Проверка совместимости схем данных

**Автоматизация**:
- При обновлении source ContentNode, автоматически триггерит:
  - Re-execute всех downstream TRANSFORMATION
  - Refresh всех связанных WidgetNode
  - Уведомления пользователей о изменениях

**Ожидаемый результат**:
- Доска становится графом обработки данных
- Данные автоматически распространяются по графу
- Пользователь видит происхождение каждого узла (data lineage)
- Трансформации автоматически переиспользуются

### FR-12: Dynamic Form Generation
**Описание**: AI-система динамически генерирует интерактивные формы ввода в процессе диалога с пользователем. Формы адаптируются под контекст, автоматически находят доступные источники данных, предлагают релевантные опции и создают ветвящуюся логику (conditional fields).
**Предусловия**: Активная сессия в AI Assistant Panel, доступ к Form Generator Agent
**Возможности**:
- Автоматическое сканирование источников данных (БД, файлы, API, Cloud storage)
- Генерация JSON schema для форм на основе контекста
- Developer Agent создаёт React компоненты форм из schema
- Умные подсказки (smart suggestions) на основе анализа данных
- Conditional logic (поля появляются/скрываются в зависимости от выбора)
- Ранжирование опций по релевантности
- Автоматический поиск недостающих данных
**Типы форм**:
- Data Source Selection: выбор источника данных (PostgreSQL, CSV, Google Drive, API)
- Analysis Options: выбор метрик и типов анализа
- Date Range Picker: выбор периода с умными пресетами
- Conditional Forms: ветвящиеся формы с динамическими полями

**Два режима работы**:
- **Auto-detect Mode**: AI сканирует доступные источники (БД, файлы, облако), показывает список найденных
- **Manual Mode**: Пользователь явно указывает источник (PostgreSQL хост, таблица, учётные данные), система валидирует подключение

**Шаги (Auto-detect)**:
1. Пользователь задаёт запрос: "Проанализируй продажи за 2025"
2. Intent Analyzer определяет: автопоиск нужен (источник не указан)
3. Data Source Scanner сканирует доступные источники

### FR-14: Widget Transformation System
**Описание**: Система интеллектуальных трансформаций виджетов. Пользователь выделяет один или несколько виджетов на доске, ИИ анализирует их содержимое и предлагает операции обработки (аналитические, композиционные, предиктивные). Результат — новый производный виджет, связанный с исходными через граф зависимостей (data lineage).

**Назначение**: Превратить GigaBoard из статического дашборда в **интерактивную аналитическую среду**, где данные трансформируются через цепочки операций.

**Основные компоненты**:

**1. UI/UX Flow**:
```
[Выделение виджетов] → [Shift+Click или Lasso] 
     ↓
[Меню трансформаций] → [Правый клик → "Трансформировать" или Ctrl+T]
     ↓
[AI предложения] → ИИ анализирует типы данных и контекст:
     • ⚡ Рекомендуемое (3-5 операций)
     • 📊 Аналитика (статистики, прогнозы, аномалии)
     • 🔄 Трансформации данных (агрегация, join, pivot)
     • 📝 Генеративные (отчёты, описания)
     • ✍️ Свой запрос (промпт пользователя)
     ↓
[Preview] → Показ результата до применения
     ↓
[Применение] → Новый виджет создаётся на доске с связями lineage
```

**2. Типы трансформаций**:

**Аналитические (Data-to-Insight)**:
- Таблица → "Построить временной ряд по колонке X", "Рассчитать корреляционную матрицу"
- График → "Добавить линию тренда", "Forecast на N периодов", "Выделить аномалии"
- 2+ графика → "Сравнить на одной оси", "Найти корреляцию"

**Композиционные (Widget-to-Widget)**:
- Таблица + График → "Создать интерактивный дашборд с фильтрацией"
- Несколько графиков → "Объединить в сводную панель", "Сравнительный анализ"
- Текст + Данные → "Создать отчёт с визуализациями"

**Трансформации данных (Data-to-Data)**:
- Таблица → "Агрегировать по месяцам", "Транспонировать", "Pivot table", "Удалить выбросы"
- 2 таблицы → "JOIN по ключу", "CONCAT", "Найти различия"
- Датасет → "Нормализовать", "Feature engineering"

**Генеративные (Insight-to-Content)**:
- График → "Написать описание тренда"
- Таблица → "Создать executive summary"
- Несколько виджетов → "Создать презентацию", "Написать отчёт"

**Предиктивные (Data-to-Prediction)**:
- Временной ряд → "Прогноз на N периодов (ARIMA, Prophet)"
- Табличные данные → "ML-модель для предсказания Y"
- Исторические данные → "Обнаружить аномалии"

**3. Data Lineage (Граф происхождения)**:
```
[Widget A: Raw Data] ──┐
                       ├──> [Widget C: Correlation Matrix]
[Widget B: Raw Data] ──┘        │
                                └──> [Widget D: Heatmap]
                                          │
                                          └──> [Widget E: Summary Report]
```
- Визуальное отображение связей между исходными и производными виджетами
- Возможность откатиться к любому этапу трансформации
- Просмотр кода трансформации для каждой связи

**4. Transformation Agent** (9-й агент в системе):
```python
class TransformationAgent:
    """
    Анализирует выделенные виджеты и выполняет трансформации.
    """
    
    def analyze_selection(self, widgets: List[Widget]) -> List[TransformOperation]:
        """Определяет типы данных, схемы, контекст и генерирует список операций"""
        
    def suggest_operations(self, context: SelectionContext) -> List[Suggestion]:
        """Умные предложения на основе типов виджетов, структуры данных, 
           истории действий пользователя и типовых паттернов"""
        
    def execute_transform(self, operation: str, inputs: List[Widget]) -> Widget:
        """Генерирует код → выполняет в Executor → создаёт виджет → 
           сохраняет lineage"""
```

### FR-17: Smart Data Lineage & Impact Analysis
**Описание**: Система отслеживания происхождения данных (data lineage) и анализа влияния изменений (impact analysis). Визуализирует полный граф зависимостей между источниками данных, трансформациями и виджетами. При изменении source data показывает какие виджеты будут затронуты. Включает AI-powered инструменты для root cause analysis и автоматического исправления проблем.

**Возможности**:
- **Lineage Graph**: Визуальный DAG (направленный ациклический граф) всех зависимостей
- **Impact Analysis**: "Если я изменю эту таблицу, какие виджеты сломаются?"
- **Root Cause Debugging**: AI анализирует сломанный виджет и показывает источник проблемы
- **Data Provenance**: История изменений данных с timestamp'ами и авторами
- **Automatic Refresh**: При изменении source виджеты обновляются автоматически
- **Break Detection**: Уведомления когда меняется schema источника ("колонка 'price' удалена")

**Визуализация lineage**:
```
[PostgreSQL: sales_table] ──┐
                            ├──> [Transformation: aggregate_by_month] ──┐
[CSV: customers.csv] ───────┘                                           ├──> [Widget: Sales Chart]
                                                                        │
[API: currency_rates] ─────────────────────────────────────────────────┘
```

**Сценарий использования (root cause)**:
```
Пользователь: "Виджет 'Monthly Revenue' показывает нулевые значения"

AI: "Анализирую проблему... 🔍
     
     Найдена причина:
     [PostgreSQL: sales_table] ❌ Connection failed
         │
         └──> [Transformation: calculate_revenue] ⚠️ No data
                  │
                  └──> [Widget: Monthly Revenue] ❌ Empty result
     
     Проблема: Соединение с PostgreSQL потеряно (credentials expired)
     
     Решение:
     • Обновите credentials в Data Sources
     • Или используйте кэшированные данные (последнее обновление: 2 часа назад)"

Пользователь: "Используй кэш"

AI: "✅ Виджет восстановлен с кэшированными данными"
```

**Технические компоненты**:
- Таблицы: DataLineageEdge, DataSource, DataTransformation, LineageHistory
- Lineage Tracker: автоматическое отслеживание всех операций с данными
- Impact Analyzer: BFS/DFS по графу зависимостей
- AI Root Cause Agent: анализ ошибок и предложение решений

**Ожидаемый результат**:
- Прозрачность: всегда понятно откуда данные и как они обработаны
- Быстрое debugging: AI автоматически находит причину проблемы
- Предотвращение ошибок: impact analysis перед изменениями

### FR-23: Source-Content Node Architecture (Архитектура Source-Content)
**Описание**: Фундаментальная архитектура разделения источников данных (SourceNode) и обработанного контента (ContentNode). Эта концепция заменяет универсальный DataNode на явное разделение "откуда данные" (Source) и "что получилось" (Content), обеспечивая прозрачность происхождения данных, автоматическое обновление и поддержку потоковых данных.

**📍 См. архитектуру**: [ARCHITECTURE.md](ARCHITECTURE.md)

**Ключевая идея**:
```
SourceNode (источник, наследует ContentNode) ──TRANSFORMATION──> ContentNode (результат) ──VISUALIZATION──> WidgetNode (визуализация)
```

> **Примечание**: SourceNode наследует ContentNode, поэтому EXTRACT edge удалён. Данные хранятся напрямую в SourceNode.

**Типы SourceNode** (9 типов):
1. **csv**: CSV файлы
2. **json**: JSON файлы
3. **excel**: Excel файлы
4. **document**: PDF, Word, текстовые документы
5. **api**: REST API endpoints
6. **database**: SQL запросы (PostgreSQL, MySQL, SQLite)
7. **research**: AI-генерация данных из текстового промпта
8. **manual**: Ручной ввод данных
9. **stream**: WebSocket/SSE потоки данных в реальном времени

**Структура ContentNode**:
- **text**: Текстовое описание (Markdown/HTML)
- **tables**: Массив таблиц [Table1, Table2, ..., TableN]
- Каждая таблица: `name`, `schema`, `rows` (JSON массив)
- Один ContentNode может содержать N таблиц

**Операции SourceNode**:
- **extract**: Извлечь данные из источника → сохранить в SourceNode (text + tables)
- **validate**: Проверить подключение/валидность источника
- **refresh**: Принудительно обновить данные
- **archive**: Сохранить активные данные в архив (для stream)

**Streaming Support** (уникальная возможность):
- **Accumulation Mode**: Новые данные добавляются к существующим
- **Archive Strategy**: `max_active_rows` → старые строки в архив
- **Update Modes**: `interval` (auto) или `manual` (by button)
- **5 Replay Modes** для трансформаций:
  1. **throttled**: Replay с задержками (эмуляция реального времени)
  2. **batched**: Порциями по N строк
  3. **manual**: По кнопке пользователя
  4. **intelligent**: AI определяет оптимальный режим
  5. **selective**: Только ключевые изменения

**Миграция от DataNode**:
- ✅ Завершена: DataNode заменён на SourceNode + ContentNode
- SourceNode наследует ContentNode (без EXTRACT edge)
- Таблицы БД: `source_nodes`, `content_nodes`, `widget_nodes`, `comment_nodes`

**Преимущества**:
- **Прозрачность**: Всегда видно откуда данные (source traceability)
- **Автообновление**: API/DB/Stream источники обновляются автоматически
- **Streaming**: Поддержка реального времени (IoT, финансы, мониторинг)
- **N таблиц**: ContentNode может содержать несколько связанных таблиц
- **Data Lineage**: Явный граф: Source → Content → Widget

**Пример использования** (IoT датчик температуры):
```
1. Пользователь: "Подключи датчик температуры из WebSocket"

2. AI создаёт:
   - SourceNode (type: stream, url: ws://sensor.example.com)
   - SourceNode наследует ContentNode → данные хранятся напрямую

3. Данные поступают в реальном времени:
   - Каждое новое измерение добавляется в SourceNode.tables[0]
   - При достижении max_active_rows=1000 → старые строки в архив

4. Пользователь: "Построй график температуры"

5. AI создаёт WidgetNode (line chart) → VISUALIZATION edge

6. При новых данных в SourceNode → WidgetNode автообновляется

7. Пользователь: "Рассчитай среднюю температуру за час"

8. AI создаёт:
   - TRANSFORMATION (SourceNode → новый ContentNode с агрегацией)
   - Replay mode: intelligent (AI сам решает как часто пересчитывать)
```

**Интеграция с другими FR**:
- FR-1: 4 типа узлов (Source, Content, Widget, Comment)
- FR-2: CRUD операции для Source/Content
- FR-3: Real-time streaming через SourceNode (type: stream)
- FR-9: WidgetNode визуализирует ContentNode/SourceNode
- FR-10: Агенты управляют Source/Content/Widget
- FR-11: TRANSFORMATION, VISUALIZATION, COMMENT edges (EXTRACT удалён)
- FR-13: Public Data Discovery создаёт SourceNode для внешних источников

**Технические требования**:
- Database schema: `source_nodes` (наследует `content_nodes`), `content_nodes`, `widget_nodes`, `comment_nodes`
- API endpoints: `/source-nodes`, `/content-nodes`, `/widget-nodes`, `/comment-nodes`
- WebSocket: `source:updated`, `content:updated` events
- Archive storage: S3/MinIO для хранения архивных данных
- Replay engine: Background jobs для автоматического refresh

**Предусловия**: FR-1 (Board & Node CRUD), FR-3 (Real-time), FR-11 (Connection Types)

- Поддержка потоковых данных в реальном времени
- Автоматическое обновление при изменении источника
