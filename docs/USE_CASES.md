# AI Assistant Panel — Примеры использования (Use Cases)

**Архитектура**: Data-Centric Canvas с SourceNode/ContentNode/WidgetNode/CommentNode

> **Примечание**: Все примеры используют новую архитектуру Data-Centric Canvas, где:
> - **SourceNode** содержит источники данных (из БД, API, файлов)
> - **ContentNode** содержит обработанные/трансформированные данные
> - **WidgetNode** содержит визуализации (генерируются AI из ContentNode)
> - **CommentNode** содержит комментарии и аннотации
> - **Edges** связывают узлы (TRANSFORMATION, VISUALIZATION, COMMENT)

## 1. Быстрый анализ метрик

### Сценарий
Product Manager открывает борд "Sprint Metrics" с текущими KPI. На доске есть **SourceNode/ContentNode** с метриками и три **WidgetNode** визуализации: retention, DAU, ARPU. Ему нужно быстро понять, есть ли тревожные сигналы.

### Диалог

```
User: "Есть ли какие-то проблемы с метриками на этой неделе?"

AI: "Я анализирую ContentNode с метриками и вижу три WidgetNode визуализации:
- DAU: 12,500 (↓ 3% vs неделя назад) ⚠️
- Retention: 42% (↓ 2%) ⚠️
- ARPU: $4.32 (↑ 1%) ✓

Два снижения. Падение DAU может быть связано с сезонностью или 
новым релизом. Стоит ли мне создать трансформацию для когортного анализа?"

User: "[✓ Добавить когортный анализ]"

AI: "Готово! Я создал:
1. TRANSFORMATION edge для группировки по когортам
2. Новый ContentNode с когортными данными
3. WidgetNode с таблицей когорт

Теперь видно, какие когорты теряют активность. Проверьте когорту 
с 10-16 января — у неё самое низкое удержание."

User: "Спасибо. А что с ARPU?"

AI: "ARPU растёт на 1% — хороший знак. Это может означать:
- Новые юзеры с более высокой стоимостью
- Улучшенная монетизация
Хотели бы вы увидеть ARPU по сегментам пользователей?"

User: "[+ Добавить ARPU по сегментам]"
```

**Результат**: PM получил полную картину проблем за 2 минуты. На доске появились:
- Новый ContentNode с когортными данными (через TRANSFORMATION)
- WidgetNode с таблицей когорт (через VISUALIZATION edge)
- Автоматическая связь данных - изменения в исходном SourceNode обновят когортный анализ

## 2. Исследование корреляции

### Сценарий
Data Scientist работает над гипотезой: "Влияет ли количество email-кампаний на конверсию?" На доске два **SourceNode**: данные email-кампаний и данные конверсии, плюс два **WidgetNode** с графиками. Ему нужно быстро проверить корреляцию.

### Диалог

```
User: "Есть ли корреляция между количеством emails и конверсией?"

AI: "Анализирую ваши два SourceNode...
✓ Положительная корреляция: r = 0.72 (сильная)

На WidgetNode визуализациях видно:
- Когда emails ↑, конверсия ↑ спустя 2-3 дня
- Когда emails ↓, конверсия ↓ (через неделю)

Это может означать причинно-следственную связь OR совместное 
влияние третьего фактора (например, маркетинговый цикл).

Рекомендация: создать TRANSFORMATION для lag-анализа (с задержкой 1, 2, 3 дня)?"

User: "[+ Добавить lag-анализ]"

AI: "Готово! Я создал:
1. Python трансформацию для расчета корреляций с разными лагами
2. Новый ContentNode с результатами lag-анализа
3. WidgetNode с таблицей корреляций:

- Lag 0: r = 0.72 ✓ (самая сильная)
- Lag 1: r = 0.65 ✓
- Lag 2: r = 0.58 ✓
- Lag 3: r = 0.42 ✓

Вывод: эффект проявляется сразу же (в день отправки)."

User: "Perfect! Покажи это руководству в презентации. 
Можешь сгенерировать резюме этого анализа?"

AI: "Готово! Я создал текстовый блок с выводами:
'Анализ показывает сильную положительную корреляцию (r=0.72) 
между объемом email-кампаний и конверсией. Эффект проявляется 
немедленно (lag-0), что свидетельствует о прямой причинно-следственной связи.'"

[✓ Добавлено на доску]
```

**Результат**: Вместо часов на Python анализ, Scientist получил результаты за 5 минут. На доске виден **граф data lineage**:
```
SourceNode(emails) ──┐
                     ├─> TRANSFORMATION ─> ContentNode(lag-analysis) ─> WidgetNode(correlation table)
SourceNode(conversion)┘
```
Любое изменение в исходных данных автоматически пересчитает lag-анализ!

## 3. Совместная доска с чатом для команды

### Сценарий
Команда аналитики (5 человек) работает над Q1 бюджетным прогнозом. На доске несколько **ContentNode** с сценариями (conservative, realistic, optimistic) и **WidgetNode** с визуализациями. Нужно быстро выбрать лучший.

### Диалог

```
Anna (Аналитик 1): "У нас три сценария. Какой самый вероятный?"

AI: "Я анализирую три ContentNode с прогнозами и соответствующие WidgetNode визуализации:
- Conservative: $2.1M (75% вероятность)
- Realistic: $2.8M (50% вероятность)
- Optimistic: $3.5M (20% вероятность)

Статистически, Realistic — хороший компромисс, но учтите 
исторические ошибки прогнозов вашей команды 
(среднее отклонение: ±12%).

Может быть, создать TRANSFORMATION для добавления коридора ошибок ±12% к каждому сценарию?"

Boris (Аналитик 2): "[✓ Добавить коридор ошибок]"

AI: "Готово! Я создал новые ContentNode с коридорами и обновил WidgetNode:
- Conservative: $2.1M ± $252k
- Realistic: $2.8M ± $336k (средний диапазон: $2.46M - $3.14M)
- Optimistic: $3.5M ± $420k

Даже с ошибками Realistic остается наиболее надежным выбором."

Natasha (PM): "А если мы достигнем эффективности на уровне 2024 Q4?"

AI: "Интересный вопрос! В Q4 2024 у вас была эффективность 0.95x. 
Если применить её к текущим сценариям:
- Conservative: $1.99M
- Realistic: $2.66M

Dmitry (Director): "Да, добавь. И помечай это как 'Adjusted for Q4 efficiency'."

AI: "[✓ Сценарий добавлен]"
```

**Результат**: Команда协同 работала над решением в одном месте, все видели диалог и рекомендации.

## 4. Исследование аномалий

### Сценарий
Инженер смотрит на **WidgetNode** с графом ошибок в production (созданный из **ContentNode** метрик). Вчера была ночь, когда количество ошибок выросло с 10 до 500. Нужно быстро понять, что произошло.

### Диалог

```

AI: "Я анализирую ContentNode с метриками и вижу аномалию на WidgetNode визуализации!
- Обычно: ~10 ошибок/час
- Вчера 23:00-02:00: ~250 ошибок/час (25x выше)

Это совпадает с развертыванием версии 3.2.1 в 22:45.

Рекомендации для анализа:
1. Создать ContentNode с timeline событий (deployments, alerts)
2. Создать TRANSFORMATION для группировки ошибок по типам
3. Сравнить с предыдущим развертыванием версии 3.2.0

Хотите я это сделаю?"

User: "[✓ Добавить все три]"

AI: "Готово! На доске теперь:
1. ContentNode + WidgetNode Timeline: видно, что deploy был в 22:45, ошибки выросли в 22:47
2. ContentNode (трансформация) + WidgetNode Таблица ошибок: показывает TopException: 'NullPointerException in PaymentService' (80% ошибок)
3. ContentNode Сравнение: версия 3.2.0 никогда не выдавала этих ошибок

Вывод: высокая вероятность регрессии в PaymentService на версии 3.2.1."

User: "Спасибо! Скину эту доску в #incidents канал."

AI: "Готово! Я создал резюме для канала:

Доска анализа: [ссылка]'"

[✓ Скопировано в буфер обмена]
```

**Результат**: Инженер потратил 5 минут на анализ вместо 30 минут на SQL запросы и Grafana.
