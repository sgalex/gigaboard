# Администратор и системные настройки LLM

## Executive Summary

В GigaBoard введён тип пользователя **администратор**: только он управляет настройками LLM для всей системы. Логин и пароль администратора задаются через переменные окружения. В разделе «Настройки LLM» доступны системные параметры провайдера/модели и **Playground** для тестирования LLM и запуска мультиагентного пайплайна.

---

## Роли пользователей

| Роль      | Описание |
|-----------|----------|
| `user`    | Обычный пользователь: доски, проекты, AI-ассистент. Не имеет доступа к настройкам LLM системы. |
| `admin`   | Администратор: все права пользователя + раздел «Настройки LLM» (системный провайдер/модель, тест LLM, Playground с мультиагентом). |

Роль хранится в поле `User.role` (`user` | `admin`).

---

## Задание администратора через env

- **ADMIN_EMAIL** — email учётной записи администратора (например `admin@gigaboard.local`).
- **ADMIN_PASSWORD** — пароль для входа.

Если обе переменные заданы при старте приложения:

1. Ищется пользователь с `email == ADMIN_EMAIL`.
2. Если пользователь не найден — создаётся с `username` из email (до `@`), заданным паролем и `role=admin`.
3. Если найден — у него выставляется `role=admin` и при необходимости обновляется хэш пароля по `ADMIN_PASSWORD`.

Таким образом, администратор входит в систему как обычный пользователь (логин/пароль из env), но с расширенными полномочиями.

## Системные настройки LLM

Настройки LLM строятся в **три уровня** (подробно: [LLM_CONFIGURATION_CONCEPT.md](./LLM_CONFIGURATION_CONCEPT.md)):

1. **Перечень LLM-пресетов** — администратор настраивает несколько «профилей» (GigaChat, внешний OpenAI-совместимый и т.д.) с реквизитами (ключи, base URL, модель). Реквизиты вводятся один раз на пресет.
2. **Модель по умолчанию** — одна из пресетов выбирается как системная по умолчанию для всех агентов.
3. **Выбор LLM по агенту** — для каждого агента (planner, discovery, analyst, codex и т.д.) можно опционально указать пресет из списка; если не указан — используется модель по умолчанию.

Читать и изменять эти настройки может **только администратор** (эндпоинты под `Depends(get_current_admin_user)`). Multi-Agent при вызовах LLM использует системные настройки: по умолчанию — выбранный пресет, при необходимости — привязку агента к пресету.

**GigaChat и другие LLM настраиваются только в панели администратора** (пресеты с API-ключом). Переменные окружения `GIGACHAT_*` не используются для системного LLM; при отсутствии пресета по умолчанию с указанным ключом мультиагент вернёт ошибку с подсказкой настроить пресет.

## Playground

В разделе «Настройки LLM» (доступен только администратору):

1. **Тест LLM** — проверка подключения к выбранному провайдеру/модели (один запрос в чат).
2. **Playground мультиагента** — чат с историей: каждое новое сообщение запускает полный пайплайн Multi-Agent (Orchestrator). Результат (план, ответы агентов, статус) возвращается в ответе и отображается в UI.

**Эндпоинт**: `POST /api/v1/admin/llm-playground/run` с телом:

- `prompt` — текущее сообщение пользователя;
- `chat_history` (опционально) — массив `{ "role": "user"|"assistant", "content": "..." }` (до последних 10 сообщений на бэкенде).

На бэкенде собирается **обогащённый** `user_request`: при наличии истории — блок «История диалога (от старого к новому): …», затем «Текущий вопрос пользователя:» и `prompt`. Далее вызывается `orchestrator.process_request(user_request=enriched_request, board_id="playground", user_id=current_admin_id, session_id=None, context={ chat_history? })`.

### Сравнение с Research Chat (`POST /api/v1/research/chat`)

| Аспект | Playground | Research Chat |
|--------|------------|---------------|
| Доступ | Только `role=admin` | Любой авторизованный пользователь |
| `board_id` у Orchestrator | `"playground"` | Placeholder UUID (исследование вне доски) |
| Форма `user_request` | История + «Текущий вопрос:» + сырой `prompt` | Сообщение пользователя + короткий форматный хинт в конце (таблицы, итог) |
| Ответ API | Полный объект результата оркестратора | Уже нормализованный `ResearchChatResponse` (narrative, tables, sources) |

Ядро одно и то же: Planner → Discovery → Research → Structurizer → Analyst → Reporter (и валидация по плану). Расхождение качества при **одинаковой** формулировке вопроса чаще всего связано с разным текстом `user_request` у Planner; подробнее — [AI_RESEARCH_SOURCE_IMPLEMENTATION_PLAN.md](./AI_RESEARCH_SOURCE_IMPLEMENTATION_PLAN.md) §2.0.1.

## Зависимости в коде

- **Роль**: `User.role`, миграция, возврат `role` в `UserResponse` и в JWT/сессии при необходимости.
- **Проверка прав**: `get_current_admin_user` (получить текущего пользователя, проверить `role == "admin"`, иначе 403).
- **Системные настройки**: перечень пресетов (`llm_config`), системные настройки (default + привязки агентов), модель `SystemLLMSettings` и связанные сущности; CRUD только для admin; `LLMRouter` при вызове LLM определяет пресет по умолчанию или по привязке агента (см. [LLM_CONFIGURATION_CONCEPT.md](./LLM_CONFIGURATION_CONCEPT.md)).
- **Playground**: роут `POST /admin/llm-playground/run` с вызовом Orchestrator, защищённый `get_current_admin_user`.

См. также: `docs/ARCHITECTURE.md`, `docs/MULTI_AGENT.md`, `docs/API.md`.
